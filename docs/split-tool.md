# Farely 设计文档

> 朋友出游/聚餐分账工具：输入名字和各项花销，自动算出"谁该转给谁多少钱"，转账次数最少。

## 1. 产品定位

- **一句话**：一个人记账，全员结清。旅行中由一个"记账员"随手录入每笔消费，行程结束一键生成结算方案，复制到群里大家照着转账。
- **不做的事（MVP 阶段）**：不要注册登录、不要好友系统、不要接支付、不要多人实时协作。这些都是后期可加的，先把"录入 → 算账 → 分享结果"这条链路做到极致顺手。

## 2. 形态选择（推荐：纯前端 PWA）

| 方案 | 优点 | 缺点 |
|---|---|---|
| **纯前端 Web/PWA（推荐）** | 零成本部署、无需后端、发个链接就能用、手机加到主屏像 App | 数据只在记账员手机上 |
| 微信小程序 | 国内分享体验最好 | 审核、账号、开发约束多，适合验证后再做 |
| 带后端的协作版 | 多人同时记账 | 复杂度翻倍，MVP 不值 |

理由：这个场景天然是"一个人记、全员看结果"，数据放本地（localStorage）完全够用。结果通过**文字摘要**或**编码进 URL 的只读链接**分享，不需要服务器。

**技术栈**：Vite + React（或 Svelte）+ TypeScript，localStorage 持久化，部署到 Vercel / Cloudflare Pages。无后端、无数据库、零运维成本。

## 3. 数据模型

```ts
interface Trip {
  id: string;
  name: string;            // "三亚五日游"
  members: Member[];
  expenses: Expense[];
  currency: string;        // MVP 单币种，默认 CNY
  createdAt: number;
}

interface Member {
  id: string;
  name: string;            // 只需要名字
}

interface Expense {
  id: string;
  description: string;     // "第一天晚饭"
  amountCents: number;     // ⚠️ 一律用"分"存整数，杜绝浮点误差
  payerId: string;         // 谁垫付的
  participantIds: string[];// 谁参与分摊（默认全员，可去掉没参加的人）
  split: SplitMode;
  createdAt: number;
}

type SplitMode =
  | { type: 'equal' }                                  // 均分（默认）
  | { type: 'weighted'; weights: Record<string, number> } // 按份数（2大1小）
  | { type: 'exact'; amounts: Record<string, number> };   // 指定金额
```

关键点：
- **金额全程用整数（分）**，只在展示层格式化成元。
- **participantIds 独立于 members**：旅行中常有"这顿饭他没来"的情况，每笔消费可勾选参与人。

## 4. 核心算法

### 4.1 算每个人的净余额

对每笔消费：付款人 `+amount`，每个参与人 `-自己的份额`。

均分时的**取整规则**（必须保证份额之和精确等于总额）：

```
base = floor(amountCents / n)
余下的 (amountCents mod n) 分，按参与人固定顺序每人多摊 1 分
```

确定性分配，谁也不会莫名其妙差一分钱。全部消费累加后得到每人净余额 `balance`（正 = 该收钱，负 = 该付钱），所有人余额之和恒为 0 —— 这是一个值得在代码里 assert 的不变量。

### 4.2 最少转账次数结清

两层策略：

**第一层：贪心（保底，≤ n−1 笔）**
欠最多的人 → 转给应收最多的人，转 `min(|欠|, |应收|)`，清零一方后继续。简单、稳定，n 个人最多 n−1 笔转账。

**第二层：子集匹配优化（真·最少笔数）**
理论上最少转账数 = `n − k`，k 是能把余额划分成的"内部恰好清零的子集"个数（这是 NP-hard 问题，但分账场景 n ≤ 15，可以暴力）。用状态压缩 DP 在 O(2ⁿ·n) 内求最大 k：

```
dp[mask] = 集合 mask 内成员能划分出的最多零和子集数
若 sum(mask) == 0: dp[mask] = max(dp[mask], max over submask s⊂mask, sum(s)==0 of dp[s] + dp[mask^s])
```

每个零和子集内部再用贪心生成具体转账。10 人以内毫秒级出结果。

实践建议：先只上贪心（对绝大多数真实账目已是最优或差 1 笔），子集 DP 作为纯函数优化项后补——两者接口一致：`settle(balances) => Transfer[]`，方便单测对比。

### 4.3 算法模块独立成纯函数

`core/` 目录下零依赖纯 TypeScript：`computeBalances(trip)`、`settle(balances)`。好处：
- 单测好写（这是全 App 最不能错的部分）；
- 将来换 UI / 做小程序，核心逻辑直接复用。

## 5. 页面与交互（3 个界面搞定）

```
┌─ 首页 ────────────┐   ┌─ 账本页（核心）──────┐   ┌─ 结算页 ─────────┐
│ 新建账本           │   │ 顶部：成员 chips +添加 │   │ 每人净余额条形图    │
│ 历史账本列表       │ → │ 流水列表（可改可删）   │ → │ 转账方案：          │
│                   │   │ 底部大按钮 [+ 记一笔]  │   │  小明 → 小红 ¥83.50│
└───────────────────┘   └──────────────────────┘   │ [复制结算文案]      │
                                                    └────────────────────┘
```

**记一笔**（弹层，10 秒内完成是设计目标）：
1. 金额（数字键盘直接弹出）
2. 谁付的（成员头像单选，默认上次的付款人）
3. 谁参与（默认全选，点头像排除）
4. 备注（可选）

**结算文案**是传播关键，一键复制成这样发群里：

```
【三亚五日游 · 分账结果】共 8 笔 ¥2,436.00
人均 ¥609.00
────────────
小明 → 老王 ¥215.50
小红 → 老王 ¥83.00
小刚 → 小李 ¥120.50
共 3 笔转清 ✅
由 Farely 生成 farely.app
```

## 6. 边界情况清单

- ✅ 某人没参加某笔消费（participantIds）
- ✅ 不均分：份数加权（带小孩）/ 指定金额（点菜各付各）
- ✅ 分不尽的零头（确定性取整，见 4.1）
- ✅ 中途加人：新成员只参与之后勾选他的消费，模型天然支持
- ✅ 有人直接转过钱了：录入一笔"还款"（payer=还钱人，participants=[收钱人]，exact 模式）即可冲账
- 🔜 多币种（旅行刚需，V2：每笔记币种 + 手动汇率，结算统一折成主币种）
- 🔜 多人协作记账（V2：加 Supabase 之类轻后端 + 分享链接同步）

## 7. 路线图

| 阶段 | 内容 |
|---|---|
| **MVP（1~2 周）** | 单账本、加成员、记账（均分+排除人）、贪心结算、复制文案、localStorage |
| V1.1 | 加权/指定金额分摊、子集 DP 最优结算、历史多账本、PWA 离线 |
| V2 | 多币种、URL 只读分享、导出图片账单 |
| V3 | 轻后端实时协作、微信小程序 |

## 8. 测试重点

核心算法必须有单测覆盖：
1. 余额之和恒为 0（任意随机账目，property-based 测一把）
2. 取整：份额之和 == 总额，精确到分
3. 结算方案执行后所有人余额归零
4. 贪心转账数 ≤ n−1；DP 结果 ≤ 贪心
5. 边界：1 个人、2 个人、有人净余额恰好为 0、全员余额为 0
